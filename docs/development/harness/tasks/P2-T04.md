# Executable Task Definition: P2-T04

## Task Identity

- **Task ID:** P2-T04
- **Phase:** Phase 2 — Establish Executable Task Authority
- **Title:** Define rolling task-replenishment and queue-exhaustion rules
- **Task type:** requirements document
- **Predecessor dependency:** P2-T03 must be completed by merged pull-request evidence.

## Purpose

Define controlled replenishment and queue-exhaustion rules so that an active phase without an executable next task stops safely rather than silently expanding into unapproved work.

## Required Deliverable

Create `docs/development/harness/TASK-REPLENISHMENT-AND-QUEUE-EXHAUSTION-RULES.md` that defines:

- the controlled stop state when the active phase has no ready task with complete executable authority;
- when a bounded planning or task-authoring PR may be created;
- that planning work must not silently begin implementation work;
- a minimum approved-task horizon for an active implementation phase;
- the stop condition when project direction, product authority, architecture authority, or executable authority is missing; and
- the need for explicit Project Owner authorization before a new phase, new scope, or implementation work begins.

## Acceptance Criteria

- Queue exhaustion and absent executable authority are defined as stop states.
- Conditions for a bounded planning or task-authoring PR are explicit.
- Planning and task-authoring are explicitly separate from implementation work.
- A minimum approved-task horizon is defined for an active implementation phase.
- Missing project or architecture authority requires a stop rather than inference.
- The deliverable creates no Phase 3 tasks or implementation work.
- No file outside the allowed path changes.

- Check consistency with the executable-format and eligibility/completion rules.

- Runtime, CI, and script validation are not required or authorized for this documentation task.
