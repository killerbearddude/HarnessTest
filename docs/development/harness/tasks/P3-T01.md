# Executable Task Definition: P3-T01

## Task Identity

- **Task ID:** P3-T01
- **Phase:** Phase 3 — Implement Repository Enforcement
- **Title:** Implement pull-request and CI enforcement
- **Task type:** repository workflow and CI-enforcement task
- **Predecessor dependency:** The Phase 3 planning pull request that introduced `docs/development/harness/PHASE-3-TASK-REGISTER.md` must be merged.
- **Task-state authority:** GitHub merged pull-request evidence determines predecessor completion; the Phase 3 planning PR must be merged before P3-T01 may become ready. GitHub open pull-request evidence determines whether a harness task is active or awaiting human review. The merged Phase 3 register and qualifying-gate evidence determine active-phase status. This P3-T01 definition governs P3-T01 scope, deliverable, acceptance criteria, validation, and stop condition. Current GitHub PR evidence controls over stale static task-status wording. A missing or invalid required task-definition field blocks execution. Exactly one harness task or planning PR may be executing or awaiting human review at a time.

## Purpose

Implement the pull-request and CI enforcement required by the approved enforcement requirements before protected-branch enforcement is configured. The task must enforce repository control evidence without introducing game, product, architecture, or workflow behavior beyond the approved requirements.

## Allowed Paths

- `.github/workflows/**`

The task may use GitHub Actions workflow configuration under the allowed path. It may inspect existing repository validation sources without modifying them.

## Forbidden Scope

No game code; product, architecture, graph-domain, evaluator, content, diagnostics, persistence, or UI work; GitHub branch-protection or other repository settings; repository scripts; package/build configuration; direct changes to `main`; Phase 4 or later work; or changes outside `.github/workflows/**` are authorized.

- `docs/development/harness/tasks/P3-T01.md`
- `docs/development/harness/REPOSITORY-PROTECTION-REQUIREMENTS.md`

## Required Deliverable

Create or update only GitHub Actions workflow configuration needed to enforce pull-request evidence requirements. The configuration must validate, for applicable pull requests:

1. one-task identity and active-phase eligibility;
2. task dependency and authority-document evidence;
3. required pull-request evidence, including validation outcomes, tests or intentionally-not-applicable tests, known limitations, deferred work, and internal-review result;
4. changed-file scope evidence; and
5. repository-appropriate quality checks only where existing repository-native checks are available and applicable.

The workflow must fail when required evidence is missing or required validation fails. It must not invent project-specific quality commands, workflow providers, scripts, or product checks. If an approved requirement cannot be enforced from available repository evidence without an unapproved artifact, the task must report that bounded blocker rather than create a substitute script or broaden scope.

## Acceptance Criteria

- Every changed file is under `.github/workflows/**`.
- The workflow configuration directly traces to the approved pull-request and CI enforcement requirements.
- Required pull-request evidence and enforcement categories are checked or the exact unsupported requirement is reported as a blocker.
- Existing quality checks are invoked only when repository-native commands or configuration establish them; no quality command is invented.
- The deliverable does not configure branch protection, alter game or product work, or create repository scripts.
- The PR records enforcement coverage, validation evidence, limitations, and any remaining blocked requirement.

## Required Validation

- Inspect every changed workflow for alignment with the required enforcement categories.
- Validate workflow syntax and event/permission configuration using repository-native or platform-provided validation where available.
- Confirm the changed-file set is limited to `.github/workflows/**`.
- Confirm every configured required check and evidence rule is traceable to an approved authority document.
- Record unavailable platform-run evidence or unsupported requirements explicitly in the PR; do not present unavailable evidence as a pass.
- Runtime game, product, architecture, and UI validation are intentionally not applicable and not authorized.

## Stop Condition

Open one reviewable P3-T01 pull request and stop. Await human review and merged GitHub evidence. Do not approve, merge, enable auto-merge, or claim completion before that evidence exists.

## Later-Task Boundary

This task cannot select, configure, draft, plan, or start P3-T02, P3-T03, P3-GATE-01, Phase 4, or any game, product, or architecture work.
