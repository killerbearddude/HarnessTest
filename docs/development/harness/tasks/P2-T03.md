# Executable Task Definition: P2-T03

## Task Identity

- **Task ID:** P2-T03
- **Phase:** Phase 2 — Establish Executable Task Authority
- **Title:** Define task eligibility and completion derivation rules
- **Task type:** requirements document
- **Predecessor dependency:** P2-T02 must be completed by merged pull-request evidence.

## Purpose

Define the deterministic rules the harness uses to derive task state and select exactly one next eligible task from repository-native authority.

## Required Deliverable

Create `docs/development/harness/TASK-ELIGIBILITY-AND-COMPLETION-RULES.md` defining:

- the meanings and derivation of blocked, ready, active, awaiting-review, completed, and phase-gate-pending states;
- GitHub merged pull-request evidence as the sole completion authority;
- predecessor dependency satisfaction and active-phase eligibility;
- the rule that a task with incomplete executable authority is blocked;
- the rule that `Proceed with next task.` selects only one ready task; and
- the rule that an open task PR creates awaiting-review state and prevents selection of any successor or concurrent harness task.

## Acceptance Criteria

- Every required task state is defined with objective entry and exit conditions.
- Completion depends on merged GitHub PR evidence, not static prose status.
- Dependency, phase, and executable-authority requirements are explicit.
- The one-ready-task selection rule and open-PR blocking rule are explicit.
- The deliverable creates no scripts, CI enforcement, GitHub configuration, workflow files, or Phase 3 work.
- No file outside the allowed path changes.
