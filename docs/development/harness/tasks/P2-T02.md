# Executable Task Definition: P2-T02

## Task Identity

- **Task ID:** P2-T02
- **Phase:** Phase 2 — Establish Executable Task Authority
- **Title:** Define executable task-definition format
- **Task type:** requirements document
- **Predecessor dependency:** P2-T01 must be completed by merged pull-request evidence.
- **Task-state authority:** GitHub merged pull-request evidence determines predecessor completion; P2-T01 must be merged before P2-T02 may become ready. GitHub open pull-request evidence determines whether a harness task is active or awaiting human review. The active Phase 2 register and qualifying-gate evidence determine active-phase status. This P2-T02 task-definition file governs P2-T02 scope, deliverable, acceptance criteria, validation, and stop condition. Current GitHub PR evidence controls over stale static task-status wording. ## Purpose

Define the standard repository-native structure for executable task definitions used by future phases. The format must provide enough bounded authority for a later `Proceed with next task.` instruction to execute an eligible task without a new owner-supplied scope prompt.

## Required Deliverable

Create `docs/development/harness/EXECUTABLE-TASK-DEFINITION-FORMAT.md` defining:

- required metadata: task ID, phase, title, task type, predecessor dependency, and task-state authority;
- required narrative sections: purpose, allowed paths, forbidden scope, authority documents, deliverable, acceptance criteria, validation, and stop condition;
- required fields that make scope, authority, dependency, validation, acceptance criteria, and stop conditions executable without a manual scope prompt;
- the distinction between a task register, which sequences and indexes approved tasks, and an executable task definition, which authorizes bounded execution of one task; and
- the prohibition on selecting or starting a later task.
