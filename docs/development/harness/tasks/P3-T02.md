# Executable Task Definition: P3-T02

## Task Identity

- **Task ID:** P3-T02
- **Phase:** Phase 3 — Implement Repository Enforcement
- **Title:** Configure protected default-branch enforcement
- **Task type:** repository-protection configuration task
- **Predecessor dependency:** P3-T01 must be completed by merged pull-request evidence.
- **Task-state authority:** GitHub merged pull-request evidence determines predecessor completion; P3-T01 must be merged before P3-T02 may become ready. GitHub open pull-request evidence determines whether a harness task is active or awaiting human review. The merged Phase 3 register and qualifying-gate evidence determine active-phase status. This P3-T02 definition governs P3-T02 scope, deliverable, acceptance criteria, validation, and stop condition. Current GitHub PR evidence controls over stale static task-status wording. A missing or invalid required task-definition field blocks execution. Exactly one harness task or planning PR may be executing or awaiting human review at a time.

## Purpose

Configure and verify default-branch enforcement for `main` using the approved repository-protection requirements and the check evidence established by P3-T01. The task must protect repository control boundaries without changing tracked project files or creating implementation work.

## Allowed Paths

- No tracked repository file may be created, modified, or deleted.
- **External configuration surface:** GitHub repository settings, rulesets, branch-protection settings, review settings, status-check requirements, conversation-resolution requirement, merge policy, and bypass policy that apply only to the default branch `main`.

## Forbidden Scope

No tracked repository files; game code; product, architecture, graph-domain, evaluator, content, diagnostics, persistence, or UI work; GitHub Actions workflow changes; scripts; non-`main` branch policies; direct pushes to `main`; Phase 4 or later work; or unapproved repository-administration changes are authorized.

## Required Deliverable

Configure the `main` protection surface to satisfy approved requirements:

1. prohibit direct pushes to `main` and require pull requests for changes;
2. require at least one human approval and dismiss stale approvals after new commits;
3. require the applicable successful enforcement checks established by P3-T01 before merge;
4. require branch freshness through an approved current-with-`main` policy or a matching merge-queue/update policy already supported by the repository;
5. require resolution of required review conversations before merge;
6. require squash merge and disallow merge and rebase merge methods unless changed by a later explicit Project Owner decision; and
7. prohibit branch-protection bypass permission for the harness.

Where the platform cannot express an approved requirement, the task must collect exact evidence of the limitation and stop as blocked. It must not claim an equivalent setting, create a workaround outside the allowed external configuration surface, or weaken an approved control.

## Acceptance Criteria

- No tracked repository file changed.
- The live `main` protection configuration is evidenced against every required protection item.
- Required status checks reference only checks established by P3-T01 or already approved repository-native checks.
- No harness bypass permission, direct-push exception, or unapproved merge method is introduced.
- Any unavailable requirement is recorded as a precise Blocked finding rather than a pass or silent exception.
- The PR identifies every changed external configuration control and supporting evidence.

## Required Validation

- Inspect current GitHub repository and `main` protection/ruleset settings against each required deliverable item.
- Confirm P3-T01 merged evidence and identify the required check names actually available from its configuration.
- Confirm no tracked repository file changed and no external setting outside the allowed `main` protection surface changed.
- Confirm no configuration grants branch-protection bypass to the harness.
- Record platform access limits, unavailable settings, or unverifiable controls explicitly in the PR.
- Runtime game, product, architecture, and UI validation are intentionally not applicable and not authorized.

## Stop Condition

Open one reviewable P3-T02 pull request and stop. Await human review and merged GitHub evidence. Do not approve, merge, enable auto-merge, or claim completion before that evidence exists.

## Later-Task Boundary

This task cannot select, configure, draft, plan, or start P3-T03, P3-GATE-01, Phase 4, or any game, product, or architecture work.
