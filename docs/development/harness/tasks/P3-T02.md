# Executable Task Definition: P3-T02

## Task Identity

- **Task ID:** P3-T02
- **Phase:** Phase 3 — Implement Repository Enforcement
- **Title:** Record and verify Project Owner protected-default-branch configuration
- **Task type:** repository-control evidence and verification task
- **Predecessor dependency:** P3-T01 and P3-R01 must each be completed by exact merged pull-request evidence, and a valid current Project Owner protected-default-branch configuration attestation must exist.
- **Task-state authority:** GitHub merged pull-request evidence determines predecessor completion. P3-T01 and the one-time corrective task P3-R01 must each have merged before P3-T02 may become ready. A current explicit Project Owner attestation must identify every required control in this definition and assert its configuration for `main`; an absent, incomplete, contradictory, or stale attestation blocks P3-T02. GitHub open pull-request evidence determines whether a harness task is active or awaiting human review. The merged Phase 3 register and qualifying-gate evidence determine active-phase status. This P3-T02 definition governs P3-T02 scope, deliverable, acceptance criteria, validation, and stop condition. Current GitHub PR evidence controls over stale static task-status wording. A missing or invalid required task-definition field blocks execution. Exactly one harness task or planning PR may be executing or awaiting human review at a time.

## Purpose

Record and verify the Project Owner checkpoint for `main` protected-default-branch configuration without claiming that the harness configured, administered, or independently inspected GitHub controls it cannot access. Live GitHub configuration is performed manually by the Project Owner because the harness lacks branch-protection and ruleset administration capability.

## Required Deliverable

Create `docs/development/harness/P3-T02-REPOSITORY-CONTROL-EVIDENCE.md` as the sole P3-T02 tracked-file deliverable.

Before P3-T02 begins, obtain a current explicit Project Owner attestation for `main` that covers every required control below. The evidence report must record the attestation and distinguish it from independent observation.

The report must include a table with exactly these columns:

| required control | evidence source | status | verification classification | limitation or note |
| --- | --- | --- | --- | --- |

The report must include one row for each required `main` protection control:

1. pull requests required and direct pushes prohibited;
2. at least one human approval required;
3. stale approvals dismissed after new commits;
4. `Harness PR Evidence` required before merge;
5. branch freshness or an approved merge-queue equivalent required;
6. required review conversations resolved;
7. squash merge enabled;
8. merge commits and rebase merges disabled; and
9. no harness bypass permission.

The report must use these verification classifications exactly:

- **Owner-attested** — asserted by the Project Owner; not independently inspectable through the connected toolset.
- **Independently observed** — directly visible through available repository metadata or GitHub evidence.
- **Unavailable for independent inspection** — required setting cannot be inspected through the connected toolset.
- **Blocked** — required setting is absent, contradictory, or cannot be owner-attested.

The report must state clearly that:

- the harness did not configure GitHub settings;
- owner-attested controls are not represented as independently verified;
- available repository metadata confirms only the merge-method evidence unless other direct evidence is actually available; and
- the report is evidence of the Project Owner checkpoint, not proof that the connected toolset can administer or fully inspect GitHub protections.

## Acceptance Criteria

- The only tracked file changed is `docs/development/harness/P3-T02-REPOSITORY-CONTROL-EVIDENCE.md`.
- A current explicit Project Owner attestation exists and covers all nine required controls.
- The evidence report contains the required five-column table and all nine control rows.
- Every row distinguishes evidence source, status, verification classification, and limitation or note.
- Owner-attested, independently observed, unavailable-for-independent-inspection, and blocked controls are not conflated.
- The report does not claim that the harness configured GitHub settings or independently verified a control unavailable through the connected toolset.
- Available repository metadata is used only for evidence actually visible to the harness, including merge-method evidence when available.
- The P3-T02 PR contains only the evidence report and no GitHub configuration, workflow, CI, script, implementation, game, product, architecture, or later-phase change.

- Inspect available repository metadata and GitHub evidence only for controls directly visible through the connected toolset; classify all other required controls without claiming independent inspection.

- Confirm the report contains each required clarification about owner attestation, harness non-configuration, limited metadata evidence, and connected-toolset capability.
